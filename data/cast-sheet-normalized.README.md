## Record shape

Each record contains:

- `store`, `storeLabel`
- `no`, `name`, `joinedAt`, `leftAt`
- `activeStatus`
- `ngArea`, `transport`, `transportFee`
- `holyWater`, `privateP`, `rope`, `teacher`, `threeP`
- `home`, `businessHotel`, `rentalRoom`
- `privateCosplay`, `shopCosplay`
- `foreigners`, `shiftRequest`
- `notes`, `other`, `internalPortal`
- `public`, `internal`

## Capability normalization

Raw values are normalized into:

- `yes`
- `no`
- `conditional`
- `unknown`

Rules:

- `〇 / ○ / ◎ / TRUE / 有 / 可 / OK / 済` -> `yes`
- `× / ✖ / ✕ / FALSE / 無 / NG / 不可` -> `no`
- `△ / 要 / 確認 / 都度 / 未 / 解除 / 相談 / 条件` -> `conditional`
- blank -> `unknown`

## Active status

Priority:

1. If `leftAt` exists -> `left`
2. If source status contains `在籍` -> `active`
3. If source status contains `退店` -> `left`
4. If source status contains `連絡無し / 停止 / 休止` -> `inactive`
5. If none of the above and `storeName` exists -> `active`
6. Otherwise -> `unknown`

## Intended use

- Public/chatbot fact lookup should prefer the `public` subtree.
- Reception and ops tooling can use the `internal` subtree.
- Free-text notes should not be exposed directly to customers without filtering.
